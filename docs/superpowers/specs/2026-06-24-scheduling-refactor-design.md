# 调度规则收口与小组模型重构设计

## 目标
把排班规则、小组成员关系和前端展示逻辑收口到单一来源，减少 `app.py` 和 `static/index.html` 里的重复判断，让后续改规则更稳。

## 背景
当前代码可以运行，但有几个明显的维护痛点：
- 同一套替班/休息/工作量判断散落在后端多个接口里，也在前端再写了一遍。
- 小组成员关系同时存在于 `staff.group_id` 和 `groups.json.members`，实际使用和存储口径不一致。
- 辅助脚本使用了写死的绝对路径，`start.bat` 每次启动都会装 Flask。

## 设计

### 1. 后端规则层
新增一个纯函数规则模块，负责这些共享判断：
- 判断某人是否属于某小组
- 获取小组成员
- 判断小组当日是否全休
- 获取小组当日实际在岗成员
- 计算某人当日工作量
- 判断某候选人是否可替班

`app.py` 只保留路由、读写 JSON 和调用规则层，不再在三个接口里复制同样的判定逻辑。

### 2. 小组数据模型
`staff.group_id` 作为成员关系的唯一真相。

`groups.json` 继续保留小组定义，但不再作为成员关系来源。接口里可以继续返回 `member_names` 供前端展示，但这个字段应由 `staff.group_id` 计算，不应从 `members` 数组读取。

删除小组时，要同步清理受影响员工的 `group_id`，避免留下悬空引用。

### 3. 前端数据流
前端小组表优先使用 `GET /api/groups` 返回的 `member_names`，而不是本地再扫一遍 `staff`。

前端用于编辑的小组成员选择框仍然基于 `staff.group_id`，这样展示和编辑都统一到同一条数据链上。

工作量统计保持现有规则，但把“成员归属”的判断集中到 `group_id`，避免前后端口径分叉。

### 4. 维护脚本
`add_group_helpers.py` 和 `inject_group_auto_sub.py` 改成基于脚本所在目录定位项目根目录，不再依赖另一台机器的绝对路径。

`start.bat` 改成“缺少 Flask 时才安装”，避免每次启动都重复装包。

## 非目标
- 不把 JSON 存储改成数据库
- 不重做 UI 布局
- 不改排班语义本身，只收口实现

## 验收标准
- 小组编辑、删除后，员工归属和展示一致
- 自动排班、单项替班、级联休息仍保持原行为
- `groups.json.members` 不再驱动任何业务逻辑
- 脚本路径改为项目相对路径
- 各关键 Python 文件可通过语法检查
